# PM Dashboard v2.2T Executive Milestone Governance Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deliver permission-aware Executive milestone RAG and monthly updates, a compact focused-drawer editor, and an auditable in-dashboard structural approval workflow on `v2.2T`.

**Architecture:** Keep the weekly Executive timeline snapshot as the historical and PDF source, move update history and approvals into append-only collections, and route every Executive mutation through Firebase callable Functions. Shared pure modules define the role matrix, RAG aggregation, freshness, and mutation rules; `index.html` and `team-2/index.html` render the compact roadmap and focused drawer while Firestore Rules block direct client mutation.

**Tech Stack:** Static HTML/CSS/JavaScript ES modules, Firebase Web SDK 10.12.2, Firebase Auth, Firestore, Firebase Functions v2 on Node.js 20, Node.js built-in `node:test`, GitHub/Vercel preview deployment.

## Global Constraints

- Target branch is `v2.2T`; production `origin/main` must remain unchanged.
- Roles are exactly `admin`, `executive`, `pm`, `engineering`, `sales`, `bd`, and `product` after migration.
- UI label for `executive` is `Executive Owner`.
- Fixed sections are `IoE Product Portfolio`, `Customer Engagements`, and `Investors & Strategy`, each retaining a stable `sectionId`.
- PM and Engineering may view and update only IoE Product Portfolio.
- Sales, BD, and Product may view every section but update only Customer Engagements.
- Admin and Executive Owner may view and update every section.
- Only Admin and Executive Owner may directly change structure; only Executive Owner approves requests.
- RAG values are Green, Yellow, and Red; automatic summary severity is `Red > Yellow > Green`.
- Visible summary RAG must use only sections visible to the current role.
- Every RAG change requires a non-empty status update explaining the change.
- Updates are append-only; `Update due` begins after 31 days and `Overdue` after 45 days.
- Historical weeks are never rewritten; structural changes carry forward only through normal future-week carryover.
- Phase 1 has no configured approval mailbox and sends no email.
- No Executive mutation may rely on hidden UI controls as authorization.
- Existing unrelated untracked files and generated assets must not be staged.

const { onCall, HttpsError } = require('firebase-functions/v2/https');
const { getFirestore, FieldValue } = require('firebase-admin/firestore');
const core = require('./executive-milestone-core');

const REGION = 'us-central1';
const db = getFirestore();

async function getActor(transaction, request) {
  const email = String(request.auth?.token?.email || '').trim().toLowerCase();
  if (!email) throw new HttpsError('unauthenticated', 'Sign in is required.');
  const userRef = db.collection('users').doc(email);
  const userSnapshot = await transaction.get(userRef);

  return { email, role: core.normalizeRole(userSnapshot.data().role) };
}
