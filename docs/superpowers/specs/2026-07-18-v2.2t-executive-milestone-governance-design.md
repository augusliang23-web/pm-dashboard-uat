# PM Dashboard v2.2T Executive Milestone Governance Design

**Date:** 2026-07-18

**Target:** `v2.2T`

**Status:** Approved for specification

## Goal

Replace the current dense Executive milestone editor with a focused, permission-aware workflow for monthly RAG and status updates. Preserve a compact management-facing roadmap, add append-only update history, lock milestone structure behind an approval workflow, and enforce every permission on the server.

Phase 1 includes an in-dashboard Executive Owner approval inbox. Configuring a unique approval email address and sending approval emails are deliberately deferred until a later phase.

## Scope

This design covers:

1. Executive milestone role names and section permissions.
2. Item, section, and quarter RAG display and calculation.
3. Monthly item status updates and update-age reminders.
4. A compact roadmap with a focused single-item editor drawer.
5. Locked milestone structure with an in-dashboard request and approval workflow.
6. Append-only updates, approvals, and audit records.
7. Server-side authorization through Firebase Functions and Firestore Rules.
8. Compatibility with historical weeks, carryover, and PDF output.

The following are outside Phase 1:

- Approval email address configuration.
- Sending approval or decision emails.
- Approving by following an email link.
- Rewriting historical week documents.
- Automatic RAG changes based on stale updates.

## Role Model

The dashboard role set becomes:

- `admin`
- `executive`, displayed as `Executive Owner`
- `pm`
- `engineering`
- `sales`
- `bd`
- `product`

`business` is replaced directly by `sales`; no compatibility alias is required because there are no existing `business` accounts. Existing `vip` accounts must be updated once to `executive` as part of rollout so the current department leader does not lose access.

## Executive Sections

The three fixed horizontal sections are:

1. `IoE Product Portfolio`
2. `Customer Engagements`
3. `Investors & Strategy`

Each stored section retains a stable `sectionId`. Display-name changes must not replace the stable identifier.

## View and Update Permissions

| Role | IoE Product Portfolio | Customer Engagements | Investors & Strategy |
|---|---|---|---|
| Admin | View and update | View and update | View and update |
| Executive Owner | View and update | View and update | View and update |
| PM | View and update | Hidden | Hidden |
| Engineering | View and update | Hidden | Hidden |
| Sales | View only | View and update | View only |
| BD | View only | View and update | View only |
| Product | View only | View and update | View only |

An update permission allows item RAG changes and monthly status updates in the permitted section. It does not allow structural changes.

Every authenticated dashboard role may submit a structural change request only for a section that role can view. A request to add an item must also target a visible section. Hidden section names, item identities, and request history are never exposed. Only `admin` and `executive` may change structure directly. Only `executive` handles request approval or rejection in the approval inbox; Admin can make the equivalent structural change directly with an audit reason.

The client, Firebase Function, and Firestore Rules must enforce the same permission contract. Hidden controls alone are not authorization.

## RAG Model

RAG uses three values:

- Green
- Yellow
- Red

Each item stores its current RAG. Section RAG and quarter RAG are calculated automatically from the most severe visible child item:

`Red > Yellow > Green`

Calculations are permission-aware. PM and Engineering quarter summaries use only IoE Product Portfolio items. Sales, BD, and Product may see all three sections, so their quarter summaries may include all visible items even though only Customer Engagements is editable. Admin and Executive Owner see global summaries.

Admin and Executive Owner may override a calculated section or quarter RAG. An override requires a reason and records the actor, timestamp, calculated value, and replacement value. Removing an override restores automatic calculation.

Hidden-section RAG must not influence a user's visible summary or expose the presence of a hidden Red or Yellow item.

## Roadmap Reading Experience

The management-facing roadmap remains a compact section-by-quarter matrix. Each item shows only:

- its RAG indicator;
- the milestone title;
- a one-line preview of the latest status update;
- the latest update date and actor;
- an `Update due` or `Overdue` indicator when applicable.

Long status text is not rendered in full inside the matrix. Selecting an item opens a focused drawer beside the matrix on wide screens and below the selected item on narrow screens.

The focused drawer contains:

1. Milestone title, section, and quarter.
2. Green, Yellow, and Red controls when the role can update that section.
3. A monthly status update field.
4. Append-only update history in reverse chronological order.
5. A structural-change request action.
6. Save and cancel actions for authorized editors.
7. Read-only content for users without update permission.

Only one item drawer is active at a time. The matrix does not expand multiple dense forms simultaneously.

## Monthly Status Updates

Each save appends a new update record. Existing update records are immutable through the dashboard.

- A status update may be added while RAG remains unchanged.
- If RAG changes, the new status update is required and must explain the change.
- The client focuses the missing field when validation fails.
- The Firebase Function repeats the same validation before writing.
- The current item stores the latest RAG and latest-update summary for efficient matrix rendering.
- Full history is read from the append-only update collection.

Update age is advisory:

- More than 31 days since the latest update: `Update due`.
- More than 45 days since the latest update: `Overdue`.
- Users may update earlier than 31 days.
- Staleness never changes RAG automatically.

## Structural Lock and Change Requests

Structural fields are locked during routine updates:

- milestone title;
- section;
- quarter;
- order;
- add;
- delete.

PM, Engineering, Sales, BD, and Product use `Request structural change`. A request contains:

- stable milestone `itemId`;
- target week and roadmap year;
- change type: add, rename, move section, move quarter, reorder, or delete;
- structured before and proposed-after values;
- requester reason;
- requester identity and role;
- creation timestamp;
- target item version.

The source roadmap remains unchanged while a request is pending.

Executive Owner sees pending requests in an in-dashboard approval inbox. The review screen shows the exact before and after values, requester, reason, target location, and potential effects. Executive Owner may approve or reject and may add a decision note.

On approval, a Firebase Function transaction:

1. verifies Firebase Auth and current `executive` role;
2. confirms the request is still pending;
3. confirms the target item version still matches;
4. applies the proposed structured change exactly as reviewed;
5. marks the request approved and applied;
6. writes an immutable audit record.

The approved proposal is applied automatically. Approval does not temporarily unlock the item for later free-form editing.

Request states are:

- `pending`
- `approved`
- `applied`
- `rejected`
- `conflict`

If the target changed after the request was created, the transaction does not overwrite it. The request becomes `conflict` and must be resubmitted from current data.

Admin and Executive Owner may make direct structural changes without approval. Direct changes require a reason and create a `direct-change` audit containing the before and after values.

## Phase 1 Approval Inbox

Phase 1 has no email dependency. Executive Owner can open an approval inbox from the dashboard and view:

- pending request count;
- requester and request age;
- target section and quarter;
- change type;
- before and after values;
- request reason;
- approve and reject actions;
- applied, rejected, and conflict history.

Admin may inspect the audit and request history but does not need an approval action because Admin can apply a direct structural change with a reason.

## Deferred Email Phase

A later phase may add a protected global setting for one approval mailbox. The intended contract is:

- only Admin can configure or replace the address;
- the address must belong to an active `executive` account;
- the backend revalidates the account before sending;
- the email opens an authenticated dashboard review page;
- opening the email link never performs a mutation;
- approval still requires Firebase Auth, a current Executive Owner role, a visible diff, and explicit confirmation.

No email address is hard-coded in Phase 1.

## Data Architecture

Existing weekly Executive roadmap snapshots remain under:

`weeks/{weekId}.strategyLayer.executiveMilestoneTimeline`

New collections are:

- `executiveMilestoneUpdates/{updateId}` for append-only monthly updates;
- `executiveMilestoneChangeRequests/{requestId}` for structural requests and decisions;
- `executiveMilestoneAudit/{auditId}` for applied approvals, direct structural changes, and overrides.

The deferred email phase may add:

`dashboardSettings/executiveApproval`

Every milestone uses a stable `itemId` across week carryover. Update records include:

- `weekId`
- `roadmapYear`
- `itemId`
- `sectionId`
- `quarter`
- `ragBefore`
- `ragAfter`
- `statusText`
- `createdBy`
- `createdRole`
- `createdAt`

The current weekly snapshot carries the latest item values for fast rendering. Dedicated update documents prevent unbounded nested history inside a week document and preserve Firestore-safe storage.

## Server-Side Mutation Boundary

The browser must not directly write Executive milestone changes.

Firebase Functions provide separate operations for:

- adding a monthly update;
- creating a structural change request;
- approving or rejecting a request;
- applying a direct Admin or Executive Owner structural change;
- creating or removing a RAG override.

Each operation validates:

- Firebase authentication;
- the current role from the protected user record;
- section view and update permission;
- selected week identity and release state;
- stable milestone identity;
- expected item version;
- required status or structural reason;
- allowed request-state transition.

Firestore Rules must prevent client-side modification of the Executive timeline inside a week document. Other approved week-editing workflows may continue only when the Executive timeline is unchanged. Backend Admin SDK transactions apply authorized Executive mutations and write the related update or audit document atomically.

## Historical Weeks, Carryover, and PDF

Historical week documents are not rewritten. A structural change applies to the active editable week and is preserved through normal future carryover. Previously released or historical weeks retain their original roadmap snapshot.

PDF generation continues to read the selected week's Executive roadmap snapshot. It therefore cannot show a future structural change in an older report. Permission-aware PDF audience rules must be updated to use `executive`, `sales`, and `bd` rather than the retired `vip` and `business` roles.

## Concurrency and Error Handling

- Repeated approval calls are idempotent and cannot apply a request twice.
- A stale item version produces `conflict`; it never overwrites newer content.
- A role, authentication, week, or section change invalidates an open drawer or approval session.
- A RAG change without a status explanation is rejected by both client and server.
- A direct structural change without a reason is rejected.
- Invalid request-state transitions are rejected and logged.
- Failure to write the roadmap and audit atomically leaves both unchanged.
- The approval inbox remains usable without any email configuration.

## Verification

Automated verification covers:

1. Every role and section combination in the permission matrix.
2. Hidden-section rendering and server-side access rejection.
3. Permission-aware section and quarter RAG summaries.
4. Worst-child RAG calculation and reason-required overrides.
5. Append-only monthly updates and reverse chronological history.
6. Required status text when RAG changes.
7. The 31-day and 45-day reminder thresholds.
8. Structural request creation, review, approval, rejection, direct change, conflict, and idempotency.
9. Firestore Rules rejecting direct Executive timeline writes.
10. Stable item IDs through carryover.
11. Historical-week and PDF compatibility.
12. Drawer behavior on desktop and narrow screens.
13. Keyboard focus, labels, error focus, and read-only states.

Manual acceptance covers:

- scanning the compact roadmap without text overload;
- opening and closing one focused drawer at a time;
- updating an authorized section as each role;
- confirming hidden and read-only sections;
- changing RAG with a required explanation;
- reviewing monthly history;
- submitting and approving one structural request;
- observing a deliberate version conflict;
- validating the approval inbox without email configuration.

## Rollout

1. Add backend role, permission, and mutation tests.
2. Deploy a temporary migration bridge that recognizes both `vip` and `executive` as Executive Owner while keeping `business` and `sales` separate.
3. Update the existing `vip` account to `executive`, replace `business` with `sales`, and add `bd`.
4. Deploy Firebase Functions, Firestore Rules, and the new UI that use the final role contract.
5. Remove the temporary `vip` compatibility bridge after the account migration is verified.
6. Verify the compact roadmap, focused drawer, update history, approval inbox, historical weeks, carryover, PDF output, and the v2.2T preview deployment.
7. Keep email settings and notification delivery disabled until the separate email phase is designed and configured.

## Success Criteria

- Users see only the Executive sections permitted by their role.
- Users update only the sections assigned to their role.
- RAG changes always include a status explanation.
- Monthly history is append-only and readable without crowding the roadmap.
- Section and quarter RAG are accurate and do not leak hidden status.
- Structural changes cannot bypass Executive governance.
- Executive Owner can approve exact proposals from the dashboard inbox.
- Admin and Executive Owner direct changes remain auditable.
- Client-side direct Executive writes are rejected.
- Historical weeks and PDFs remain stable.
- Phase 1 operates fully without a configured email address.
