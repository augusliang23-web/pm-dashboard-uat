# v2.2T Executive milestone inline structural changes

## Goal

Replace the current form-led structural change dialog with a contextual workflow that lets a user act from the relevant milestone card or target quarter. Keep the existing role matrix, reason requirement, request/approval flow, optimistic version check, append-only request history, and Admin/Executive Owner direct-change audit trail.

## Status-update record

The dashboard separates milestone health from the freshness of its leadership status update. RAG remains the only coloured health signal. The supporting metadata line is titled `Update record`; it does not use task-state words such as `Overdue`, `Not started`, `Current`, or `Update due`.

- a recorded update shows `Update record · 12 Jul 2026`;
- a milestone with no update shows `Update record · No update recorded`;
- more than 31 days since the last record adds the neutral text `Refresh requested`;
- more than 45 days since the last record adds the neutral text `Please refresh`.

These signals use a muted gray-blue message or clock icon, not red, amber, or RAG colours. Their tooltip says: `This indicates the freshness of the leadership status update only. It does not indicate milestone delay or change the RAG status.`

## Inline interaction

### Existing milestone card

Selecting a card opens the existing focused milestone drawer. Alongside the monthly update controls, the structural action opens a compact contextual sheet for that milestone:

- `Rename` shows the existing title and a replacement title field.
- `Move` shows only destination Section and Quarter selectors, prefilled with the current location.
- `Delete` shows a plain-language confirmation.

The system derives the current item, source location, expected version, and position. The user never enters an item ID, numeric position, or JSON diff.

### New milestone

Each editable quarter cell exposes an `+ Add milestone` action. It opens a compact sheet with only:

- milestone title;
- destination Section and Quarter, prefilled from the clicked cell;
- required reason.

The new milestone is added to the end of the selected cell. Reordering is deferred; it is not exposed in this first inline version.

### Review before submission

The sheet renders a one-sentence plain-language summary, for example:

`Move “EDB program secured” from Investors & Strategy / Q1 to Customer Engagements / Q2.`

It contains a required reason and one action button:

- non-leadership roles: `Submit for approval`;
- Admin and Executive Owner: `Apply change`.

The request record retains the exact before/after payload for server validation, conflict detection, and the approval inbox. The raw JSON is hidden from the requester but remains available in the Admin/Executive audit view.

## Permissions and safety

- A user sees only inline actions for a section they may view.
- Admin and Executive Owner can apply direct changes; every direct change requires a reason and creates an audit record.
- PM, Engineering, Sales, BD, and Product submit requests according to their existing section visibility; no role receives a direct structural-edit exception.
- Released weeks remain read-only.
- Any stale version, deleted target, or changed source location becomes a conflict without overwriting the current roadmap.

## Scope

This replaces only the browser interaction for structural changes and freshness wording. The current callable contracts, request schema, approval inbox, and audit storage remain unchanged. Per-section Firestore confidentiality and active-week enforcement remain separate deployment blockers and are not changed by this UI work.

## Verification

- Unit/source tests cover new freshness labels and no longer classify a missing update as overdue.
- Both dashboard copies expose contextual add, rename, move, and delete paths without a position field or requester-visible JSON.
- Existing callable payloads still contain exact before/after snapshots and reasons.
- Role, release, conflict, approval, and audit regressions remain covered.
