# v2.2T Executive Milestone Cross-grid Drag Design

## Goal

Replace button-led milestone moves with a deliberate cross-grid drag workflow, remove the redundant legacy editor entry point, and make Executive milestone axes and access rules configurable by stable IDs rather than their displayed names.

## User experience

### Arrange milestones

- A milestone that the current role may structurally change shows one dedicated `☰` drag handle. Text remains selectable and opening the item still opens its monthly-update drawer.
- A user may drag the milestone within its current cell to reorder it, to another quarter, or to another Executive section.
- Valid destination cells show an insertion indicator. Cells the role may not use are not valid drop targets.
- Dropping does not persist a change. It opens a compact confirmation sheet that states, in readable text, the source and destination: `Move “<title>” from <section> / <quarter> to <section> / <quarter>.`
- The user must enter a reason. Admin and Executive Owner apply the move directly; other permitted roles submit the same exact proposal for Executive Owner approval.
- Rename and delete remain available in the item drawer. The old `Move to` button is removed because dragging replaces it.

### Edit the timeline structure

- The redundant `Edit Quarterly Milestones` button in the Overview header is removed.
- Adding a milestone remains contextual: `+ Add milestone` is shown in each permitted destination cell.
- Axis names are not used for authorization. A Section and Quarter each have an immutable opaque ID and a separately editable display label.
- Admin and Executive Owner can change Section and Quarter labels through a dedicated timeline-structure configuration surface. This change is structural, recorded with a reason, and has the same approval rule: Admin and Executive Owner direct; other roles request approval.
- Adding, deleting, or reordering Sections or Quarters is out of this increment. This avoids invalidating historical milestone coordinates and existing approval records. The wording may change freely without affecting access.

## Dynamic access model

- The timeline configuration stores `sections`, `quarters`, and a policy map keyed by stable IDs. A section records the roles that may view, update monthly status, and request structural changes for that section.
- The built-in roles remain `admin`, `executive`, `sales`, `bd`, `product`, `pm`, and `engineering`; no role is inferred from a displayed section or quarter name.
- `admin` and `executive` retain global structural authority. Other roles can only begin a drag, add, rename, or delete for a source and destination allowed by the configured policy.
- Client-side rendering uses the configured policy to determine visibility, drawer controls, drag handles, and valid drop targets.
- Callable Functions reload the current configuration and validate the source ID, destination ID, quarter ID, actor role, expected item version, and mandatory reason before creating or applying a proposal. The browser payload is never authoritative.
- Append-only update, request, and audit records store the IDs and copy the effective permitted roles required for their own Firestore reads. Display labels are resolved at render time from the current configuration, with a stored fallback label for deleted or renamed configuration values.

## Data and authorization boundary

- The existing `move` change type remains a single atomic proposal with `before` and `after` coordinates, including the target insertion index. It remains compatible with the approval inbox and audit log.
- The existing fixed `SECTION_IDS` and client `EXECUTIVE_SECTIONS` lists are replaced by normalized timeline configuration at both browser entry points and in the Functions core.
- The current embedded `weeks` document cannot enforce per-section Firestore reads; it can only hide restricted sections in the UI. This increment will not claim stronger database confidentiality than that storage shape provides.
- Before production deployment, section-level confidentiality requires a separate server-enforced data shape (for example one document per Executive section/period) or an explicit product decision that all users who can read a week may read its full Executive timeline.

## Validation

- Test drag eligibility, valid / invalid destination calculation, cross-section / cross-quarter payload generation, target ordering, and a stale-version rejection.
- Test that displayed Section and Quarter labels can change without altering IDs or authorization behavior.
- Test client visibility and control rendering from configuration rather than fixed names in both dashboard entry points.
- Test callable validation and audit output for a cross-grid move using current configuration.
- Test that the removed legacy overview button is absent.

## Non-goals

- No production deployment or push in this task.
- No automatic migration of existing Firestore week documents to a section-isolated storage model.
- No immediate persistence on drop; every move continues to require an explicit reason and either direct authority or approval.
